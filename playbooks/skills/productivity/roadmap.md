# Roadmap

## Purpose

Maintain `docs/_roadmap.md` — the **plan of execution** across all work. Where `docs/_active.md` is a
flat list of everything open, the roadmap adds the thing a flat list can't: **sequence and intent** —
what to do Now, what's Next, and what's Later, in the order you mean to do it.

It sits above the per-change planning skills (`planning-workflow`, `prd-to-plan` plan a single change);
the roadmap is the portfolio view that decides *which* changes happen and *when*.

## Model

- **Horizons** — `Now` (actively being worked or up next), `Next` (committed, not started), `Later`
  (someday / not yet detailed). Placement is a deliberate human decision, never auto-derived from status.
- **Items** — each line references a `T-NNN` todo or an `I-NNN` idea not yet promoted. A todo item is
  rendered as a collapsible `<details>` block so its phases/subtasks are available on demand without
  cluttering the high-level view. A raw idea is a plain one-liner (it has no phases yet).
- **Source of truth** — the todo files remain authoritative for *content* (phases, status). The roadmap
  owns only *placement and ordering*. When you refresh, re-read the todos to update the rendered detail,
  but keep each item's horizon unless the user moves it.

## Process

### Place / reprioritize

When the user wants to plan or re-sequence:

1. Read `docs/_active.md` (open + in_progress) and any `I-NNN` ideas worth surfacing.
2. With the user, decide each item's horizon and order within it. Ask about dependencies and what they
   actually intend to tackle next — don't infer the plan from priority alone.
3. Write `docs/_roadmap.md` (format below), preserving existing placements the user didn't change.

### Refresh (keep detail current without re-planning)

Re-read the todo files for items already on the roadmap and update each collapsible block's
status/phase checkboxes to match the source todo. Do **not** move items between horizons. Drop items
whose todo is `done`/`cancelled` (they live in `docs/_done.md` now) and mention them in the summary.

### Report

Summarize the plan: counts per horizon, what's at the top of Now, and any items blocked or waiting.

## Format

````md
## Now

<details><summary><b>T-051 (D) fix slow login</b> — auth · in_progress</summary>

- [x] Phase 1: profile
- [x] Phase 2: optimize query
- [ ] Phase 3: verify
</details>

<details><summary><b>T-042 (F) dark mode toggle</b> — ui · open</summary>

- [ ] Phase 1: add toggle
- [ ] Phase 2: persist
</details>

## Next

- T-053 (F) font-size control — ui

## Later

- I-007 weekly digest email (idea)
````

The `<summary>` carries the at-a-glance plan: `T-NNN (TYPE) title — area · status`. Keep a blank line
after `</summary>` so the checklist renders. Use a plain bullet (no `<details>`) for ideas and for
todos whose phases you don't want to surface.

## Relationship to other files

- `docs/_active.md` / `docs/_done.md` — derived ledgers (what's open / what's finished). The roadmap is
  the *intentional plan*; it is not regenerated by `scripts/sync-todo-ledgers.sh`.
- `docs/_inbox/` — raw ideas; promote with `/triage-inbox` before scheduling a `T-NNN` into `Now`.

## Quality bar

- Horizons reflect the user's actual intent and ordering, not just priority sort.
- Every referenced `T-NNN`/`I-NNN` exists; done/cancelled items are off the roadmap.
- Collapsed summaries are readable as a standalone plan; expanded detail matches the source todos.
